AI 서비스를 운영하며 가장 큰 고민 중 하나는 '입력 토큰 비용'과 '응답 속도'입니다. 긴 문서를 참고하거나 복잡한 시스템 지시문을 사용하는 서비스라면 이 고민은 더욱 깊어지죠. 답변 품질을 위해 문맥(Context)을 줄일 수는 없고, 비용은 감당하기 버거울 때 가장 먼저 시도해야 할 최적화 전략, 바로 프롬프트 캐싱(Prompt Caching)을 소개합니다.

## 1. 프롬프트 캐싱이란 무엇인가?

많은 분이 프롬프트 캐싱을 "이전 대화 내용을 저장했다가 재사용하는 것"으로 오해하곤 합니다. 하지만 이는 엄밀히 말하면 응답 캐싱(Response Caching)에 가깝습니다.

프롬프트 캐싱의 핵심은 **'입력의 반복되는 앞부분(Prefix)을 처리한 중간 결과값을 재사용'**하는 것입니다.

LLM은 프롬프트를 처리할 때 내부적으로 방대한 연산 과정을 거쳐 텍스트를 이해합니다. 이때 매번 같은 지시문이나 규정 문서를 처음부터 끝까지 다시 계산하는 것은 비효율적입니다. 프롬프트 캐싱은 이 '반복되는 앞부분'을 시스템 메모리에 저장해두었다가, 다음 요청 시 해당 내용을 다시 계산하지 않고 즉시 불러오는 기술입니다.

## 2. 실제 효과: Care Access의 사례

실제 현장에서는 어떤 효과가 있을까요? 의료 기록 분석 서비스인 Care Access는 Amazon Bedrock의 프롬프트 캐싱을 도입하여 다음과 같은 놀라운 성과를 거두었습니다.

- **비용 절감**: 전체 처리 비용 86% 절감
- **속도 향상**: 기록당 처리 시간 66% 단축

의료 기록처럼 길고 반복적인 문서(컨텍스트)를 기반으로 매번 새로운 질문을 던지는 구조라면, 프롬프트 캐싱은 모델 변경 없이도 즉각적인 비용 최적화를 가능하게 합니다.

## 3. 캐시 적중률을 높이는 프롬프트 설계법

프롬프트 캐싱의 효과는 '프롬프트를 어떻게 구조화하느냐'에 달려 있습니다. 핵심 원칙은 **"변하지 않는(상수) 영역은 앞에, 매번 바뀌는(변수) 영역은 뒤에"** 배치하는 것입니다.

### 나쁜 예 (캐시 적중이 어려운 구조)

매번 바뀌는 정보(날짜, 질문 등)가 앞에 있으면 모델은 매번 '새로운 입력'으로 판단하여 캐시를 활용하지 못합니다.

```
[오늘 날짜: 2026-06-24]
[질문: 휴가 신청 방법은?]
[긴 회사 규정 문서...]
```

### 좋은 예 (캐시 적중을 극대화하는 구조)

반복되는 정책과 지시문을 앞에 고정하고, 변동 데이터를 뒤로 보냅니다.

```xml
<system_policy>
[답변 원칙]
[답변 형식]
[회사 인사 규정 20페이지]
</system_policy>

<user_info>
요청별 정보:
오늘 날짜: 2026-06-24
질문: 이 규정에 따르면 휴가 신청은 어떻게 하나요?
</user_info>
```

## 4. 실무자를 위한 3단계 적용 가이드

1. **반복 영역과 변동 영역 분리**: 현재 사용 중인 프롬프트를 분석하여 고정값(시스템 지시문, 정책, 도구 정의)과 동적값(사용자 질문, 현재 시간)을 구분하세요.
2. **구조화된 템플릿 적용**: 위 예시처럼 XML 태그 등을 활용해 프롬프트를 정형화합니다.
3. **성과 측정**: 서비스 API의 Usage Metadata를 확인하여 'Cache Read Input Tokens' 항목이 발생하는지 모니터링하세요. 캐시가 실제로 적중하고 있는지 확인하는 것이 가장 중요합니다.

## 마치며

프롬프트 캐싱은 AI 서비스의 '운영 효율'을 높이는 가장 정석적인 방법입니다. 모델을 더 작은 것으로 바꾸거나 프롬프트를 억지로 줄여 품질을 떨어뜨리기 전에, 여러분의 프롬프트 구조부터 정비해보세요. 반복되는 문맥을 모델이 다시 계산하지 않게 하는 것만으로도 비용과 속도라는 두 마리 토끼를 잡을 수 있습니다.
